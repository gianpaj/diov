# Telegram/TON Coin Economy and Queue Access Plan

## Summary
- Build the economy as a backend-first account system tied to Telegram identity and wallet flows. The primary token purchase path is Telegram-native crypto wallet checkout on TON.
- Keep one `coins` balance for both earned and purchased value. The backend owns wallet, ledger, catalog, inventory, and equipped loadouts; SpacetimeDB receives only match-scoped appearance and later entitlement snapshots.
- Support three user states:
  - Anonymous users can browse the shop/customizer and view prices.
  - Anonymous users may join `guest` matches only.
  - Registered Telegram/auth users may join `competitive` and `casual_powerups` matches and use the account economy.
- Ship v1 with cosmetics only. Future “extra bullets” should be modeled as upgrades to the existing `spit` capability and restricted to `casual_powerups`, never `competitive`.

## Key Changes
- Backend commerce/auth foundation:
  - Persist account identity through the existing auth backend, with Telegram as the primary registered identity.
  - Add `wallet`, append-only `wallet_entry`, `catalog_item`, `inventory_item`, and `equipped_loadout` tables in the backend DB.
  - Add payment integration notes and interface boundaries for Telegram wallet / TON purchase flows now, but keep phase 1 implementation on dev grants and fake purchases until checkout/webhook work starts.
  - Design purchase records to store external references needed for TON transaction reconciliation later.
- Queue and access model:
  - Add canonical room modes: `guest`, `competitive`, and `casual_powerups`.
  - `guest` is the only queue available to anonymous users and has no account economy, no persistent inventory, and no registered progression.
  - `competitive` and `casual_powerups` require a registered session.
  - Home flow asks for a mode, remembers the last valid choice per user state, and downgrades anonymous users away from registered-only queues.
- Frontend UX:
  - Expose browseable shop/customizer surfaces to everyone.
  - Gate buy, equip, claim, and registered-queue entry behind sign-in with Telegram/auth.
  - Show TON/Telegram wallet as the primary future purchase method in commerce copy and product notes.
  - On registered join, fetch equipped cosmetics and include appearance selection in the join flow.
- SpacetimeDB boundary:
  - Extend `room` with `mode`.
  - Extend `player` with cosmetic render fields such as `skinId?` and resolved display color.
  - Keep balances, purchase history, payment state, and inventory out of SpacetimeDB.
  - In v1, reducers accept cosmetic appearance only. No paid combat advantage is active yet.
- Phase 2 combat economy:
  - Add backend-owned combat entitlements such as `SPIT_AMMO_TIER`.
  - Apply them only in `casual_powerups`.
  - Sync the allowed match entitlement snapshot into SpacetimeDB before a match starts and have reducers enforce it authoritatively.

## Public Interfaces
- Backend endpoints:
  - `GET /api/user/tokens`, `GET /api/user/tokens/history`, `POST /api/daily-claim`
  - `GET /api/shop/items`, `POST /api/shop/items/:id/buy`
  - `GET /api/user/skins`, `POST /api/user/skins/:id/equip`
  - Future purchase initiation endpoint should be Telegram/TON-first and return the data needed to launch wallet payment and reconcile completion.
- SpacetimeDB/public row changes:
  - `room.mode: 'guest' | 'competitive' | 'casual_powerups'`
  - `player.skinId?: string`
  - `player.color` remains the resolved render color used by the client

## Test Plan
- Backend:
  - Anonymous users can browse catalog endpoints but cannot claim, buy, equip, or access registered-only APIs.
  - Ledger remains append-only and idempotent for rewards, grants, and purchases.
  - Catalog purchases and equips reject unavailable or unowned items.
  - TON purchase records can be created with stable external reference ids for later reconciliation.
- Frontend:
  - Anonymous users can browse commerce UI and are redirected to sign-in when attempting restricted actions.
  - Anonymous users can enter `guest` only, not `competitive` or `casual_powerups`.
  - Registered users retain last-used registered queue and see equipped cosmetics in lobby/match.
- SpacetimeDB:
  - Room mode is reflected correctly in subscriptions and UI.
  - Guest matches ignore account-bound cosmetics/entitlements beyond default presentation.
  - Future phase-2 tests verify casual-only combat entitlements and rejection in `competitive`.

## Assumptions
- Telegram identity and Telegram-native TON wallet are the primary registered-account and purchase path.
- Phase 1 still excludes real checkout/webhook implementation, but the data model must be ready for TON transaction references and reconciliation.
- Anonymous users do not accumulate persistent coins or inventory.
- V1 ships `SKIN` and `COLOR_PACK`; trails and combat upgrades follow on the same catalog framework.
